# OAuth2 구현

---

## 1. OAuth2란?

Open Authorization 2.0의 축약어로, 사용자의 비밀번호를 직접 받지 않고  
구글 / 카카오 같은 외부 서비스를 통해 사용자를 인증(Authentication)하는 표준 방법.

> 예시: GitHub에서 구글 계정으로 로그인 버튼을 눌렀을 때 일어나는 과정

회원가입(Sign Up) 시에는 username/email + 비밀번호를 입력하므로  
회원 정보는 웹 사이트 DB에 존재하지만, 로그인은 구글을 통해 진행한다.

---

## 2. 왜 필요한가?

**기존 방식의 문제점**
- 사용자가 사이트에 username/password 입력
- 서버가 구글에 대신 로그인 시도
- → 사용자 비밀번호가 서버에 노출되는 구조

**OAuth2 방식**
- 구글 로그인 페이지에서 사용자가 직접 로그인
- 서버는 **허가 코드(Authorization Code)** 를 받음
- 허가 코드로 사용자 정보 획득
- → 구글만 구글 비밀번호를 확인할 수 있음

### 인증 흐름

```
사용자가 구글 페이지에서 직접 로그인
  → 구글이 Authorization Code 발급
    → Authorization Code로 Access Token 획득
      → Access Token으로 구글 API에서 사용자 정보 수신
        → 서버가 사용자 정보 확인 후 웹사이트 JWT 발급
          → JWT를 프론트로 전달
```

### application.properties

```properties
# MariaDB 설정
spring.datasource.url=jdbc:mariadb://localhost:3310/oauth2_demo
spring.datasource.username=root
spring.datasource.password=1234
spring.datasource.driver-class-name=org.mariadb.jdbc.Driver

# Hibernate / JPA 설정
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MariaDBDialect

# GCP OAuth2 설정
spring.security.oauth2.client.registration.google.client-id=MY_CLIENT_ID
spring.security.oauth2.client.registration.google.client-secret=MY_CLIENT_SECRET
spring.security.oauth2.client.registration.google.scope=email,profile
spring.security.oauth2.client.registration.google.redirect-uri=http://localhost:8080/login/oauth2/code/google

spring.security.oauth2.client.provider.google.authorization-uri=https://accounts.google.com/o/oauth2/v2/auth
spring.security.oauth2.client.provider.google.token-uri=https://oauth2.googleapis.com/token
spring.security.oauth2.client.provider.google.user-info-uri=https://www.googleapis.com/oauth2/v3/userinfo
# sub: 구글이 부여하는 사용자 고유 식별자
spring.security.oauth2.client.provider.google.user-name-attribute=sub

# JWT 설정
jwt.secret=mySecretKeyForJwtTokenGenerationMustBe256BitsLong1234567890
jwt.expiration=86400000
```

> `jwt.expiration=86400000` → 밀리초 단위. 86400000ms = 24시간

### Entity 설계

**User / OAuth2User를 별개 엔티티로 분리하는 이유**

하나의 User 테이블에 provider 필드를 두면:
- 소셜 로그인 연동이 여러 개인 경우 → provider 값이 여러 개가 됨
- 연동하지 않은 경우 → provider가 null

이를 해결하기 위해 **테이블을 두 개로 분리하고 정규화**한다.

```
users 테이블
└── id (PK)

oauth2_users 테이블
└── user_id (FK → users.id)
└── provider (google / kakao 등)
```

> 한 명의 User(id=100)가 구글/카카오 두 곳 모두 연동하면  
> `oauth2_users`에 `user_id=100`인 row가 두 개 생성된다.

### Repository

- JPA 기반으로 작성
- `Optional` 사용 이유: `NullPointerException` 처리를 강제하지 않기 위해  
  단, 값을 꺼낼 때 `.get()` 또는 `.orElseThrow()` 호출이 필요하다.
- 복수의 특정 필드를 사용하는 커스텀 메서드를 정의한 이유: **보안상**

---

### DTO

로그인 / 회원가입 / OAuth2 로그인마다 필요한 필드가 다르므로 DTO를 분리한다.

| 상황 | 필요한 필드 |
|------|-----------|
| 회원가입 | 전체 필드 |
| 일반 로그인 | email + password |
| 구글 로그인 | password 없이 구글 사용자 정보만 |

> `carfront`에서 `AccountCredential`을 `record`로 정의했었는데,  
> 이번에는 `class`로 정의한다.

**Validation 의존성 추가 이유**  
DTO 작성 시 `@NotNull`, `@Email` 등의 어노테이션으로  
필요한 필드 유효성 검사를 Spring Boot 레벨에서 처리하기 위해.

## 4. 자주 발생하는 오류 & 실수 모음

### GCP / OAuth2 설정 관련

**리디렉션 URI 불일치**
```
redirect_uri_mismatch 오류
```
- GCP에 등록한 리디렉션 URI와 `application.properties`의 URI가 정확히 일치해야 한다.
```properties
# GCP 등록값과 아래 값이 완전히 동일해야 함
spring.security.oauth2.client.registration.google.redirect-uri=http://localhost:8080/login/oauth2/code/google
```

**scope 미설정**
```properties
# ❌ scope 없으면 기본값으로 동작해서 원하는 정보를 못 받을 수 있음
# ✅ email, profile 명시
spring.security.oauth2.client.registration.google.scope=email,profile
```

### application.properties 관련

**client-id / client-secret 깃허브에 노출**
- 민감한 키값은 절대 커밋하면 안됨
- `application.properties`를 `.gitignore`에 추가하고 `application.properties.example`을 따로 커밋할 것
```gitignore
# .gitignore
application.properties
```

**user-info-uri 오타**
```properties
# ❌ 오타 주의
spring.security.oauth2.client.provider.google.user-info-uri=https://www.googeapis.com/oauth2/v3/userinfo

# ✅ 정확한 URL
spring.security.oauth2.client.provider.google.user-info-uri=https://www.googleapis.com/oauth2/v3/userinfo
```

### Entity 설계 관련

**User 테이블에 provider 필드를 직접 추가하는 경우**
- 소셜 로그인 연동이 여러 개면 provider 값이 여러 개가 되고,  
  연동하지 않은 경우 null이 들어가는 등 테이블이 지저분해진다.
- 반드시 `User` / `OAuth2User` 테이블을 분리하고 FK로 참조하도록 정규화할 것.

---

### DTO 관련

**Entity를 직접 요청/응답에 사용하는 경우**
- Entity를 그대로 쓰면 불필요한 필드까지 노출되거나 전달될 수 있음.
- 로그인 / 회원가입 / 응답 각각 목적에 맞는 DTO를 분리해서 사용할 것.

| 클래스 | 용도 | 주요 필드 |
|--------|------|----------|
| `SignupRequestDto` | 회원가입 요청 | email, password, username 등 전체 필드 |
| `LoginRequestDto` | 로그인 요청 | email, password |
| `AuthResponseDto` | 인증 응답 | JWT 토큰 |
